## 一、项目定位

**一路记 WayLog** 是一款个人旅行助手 App，定位为「旅行控制台 + 旅行手账」，而非旅游社区或营销平台。核心目标是解决旅行信息分散、行程难整理、旅途中难查看的痛点，为个人用户提供一站式的旅行规划与管理体验。

- **Bundle ID**: `com.travelapp.mobile`
- **App 名称**: 一路记
- **Slug**: travel-app
- **版本**: 1.0.0
- **阶段**: MVP 迭代阶段

## 三、核心功能（当前已实现）

- ✅ 三种行程状态：计划中 / 旅途中 / 已完成

### 3.2 按天行程

- ✅ 地点照片支持（来源：高德、Unsplash 等）

### 3.4 预算与记账
- ✅ 行程预算设置
- ✅ 费用记录（TripExpense）：标题、金额、分类、日期、关联地点
- ✅ 7 种费用分类：交通 / 住宿 / 餐饮 / 门票 / 购物 / 活动 / 其他
- ✅ 多币种支持（CNY 等）

### 3.5 交通与住宿
- ✅ 交通安排：航班 / 火车 / 自驾 / 巴士 / 其他
- ✅ 住宿安排：名称、地址、入住/退房时间

### 3.6 备忘与清单
- ✅ 旅行备忘（TripMemo）：支持置顶
- ✅ 打包清单（TripChecklistItem）：可勾选完成状态

### 3.7 用户认证
- ✅ 4 种登录方式：游客 / 邮箱验证码 / 手机验证码 / 微信

- ✅ 微信登录通过 Edge Function `wechat-login` 实现

### 3.8 云同步

### 3.9 地图功能

- ✅ 地图调起：高德 > 苹果地图 > 百度地图 > 系统兜底

## 四、数据模型

### 核心聚合根：Trip（行程）

```
Trip
├── TripDay[]              → 按天组织的日程
│   └── TripDayItem[]      → 单个行程点（时间/地点/备注/花费/推荐理由）
├── TripPlace[]            → 地点库（名称/分类/坐标/图标/照片/评分/详情）
├── TripTransport[]        → 交通安排（航班/火车/自驾/巴士）
├── TripLodging[]          → 住宿安排（入住/退房/地址）
├── TripMemo[]             → 备忘（标题/详情/置顶）
├── TripChecklistItem[]    → 打包清单（标题/完成状态）
├── TripExpense[]          → 费用记录（分类/金额/日期/关联地点/支付人）
├── TripBudget             → 预算（金额/货币）
├── TripImportSource[]     → 导入来源（链接/文字/图片，解析待实现）
└── routeModeOverrides     → 路线交通方式手动覆盖
```

### 云端表结构（Supabase）

| 表名 | 用途 | 关键字段 |
| :-- | :-- | :-- |
| `public.profiles` | 用户资料 | display_name, avatar_url, bio, home_city |
| `public.user_trips` | 旅行数据 | user_id, payload(JSONB), version, deleted_at |
| `public.user_favorite_places` | 收藏地点 | user_id, payload(JSONB), provider_place_id, version, deleted_at |
| `public.user_route_preferences` | 路线偏好 | user_id, payload(JSONB) |
| `public.user_expense_preferences` | 费用偏好 | user_id, payload(JSONB) |

> 所有表均启用 RLS（Row Level Security），通过 `auth.uid() = user_id` 进行数据隔离。

## 五、架构设计

### 5.1 分层架构

```
app/          → 页面编排、路由、屏幕级交互（Expo Router 文件路由）
components/   → UI 呈现、可复用视图（不写业务逻辑，纯展示组件）
features/     → 业务规则、数据存储、API 调用、纯逻辑（不依赖 React，可独立测试）
hooks/        → 应用级 React hooks
```

### 5.2 数据流

```
UI (app/) → storage.ts (CRUD) → AsyncStorage (本地优先写入)
                               → cloud-sync.ts (后台异步同步) → Supabase
```

- **本地优先**：所有写操作先写入 AsyncStorage
- **脏标记**：写入后标记 dirty，触发后台同步
- **增量同步**：使用 upsert + 软删除，失败保留 dirty 标记下次重试

### 5.3 路由结构

```
RootLayout
└── RootNavigation (Stack)
    ├── (tabs)                → 底部 Tab 导航
    │   ├── index             → 行程列表（首页）
    │   ├── create            → 新建行程（+号）
    │   └── profile           → 我的
    ├── account               → 账号与安全
    ├── feedback              → 用户反馈
    ├── preferences           → 偏好设置
    ├── search                → 全局搜索
    ├── trips/new             → 新建行程
    ├── trips/[id]            → 行程详情（动态路由）
    ├── trips/edit            → 编辑行程
    ├── trips/place           → 地点详情
    ├── trips/favorite-place  → 收藏地点搜索
    ├── trips/favorite-places → 收藏地点列表
    └── trips/import          → 导入攻略
```

## 七、开发约定

1. **分层严格**：`app/` 只管页面编排，`components/` 只管 UI 呈现，`features/` 管业务逻辑和数据
2. **features 不依赖 React**：业务逻辑写成纯函数，可独立测试
3. **新增功能流程**：先在 `features/` 写逻辑和测试 → 再在 `app/` 和 `components/` 接入
4. **移动端优先**：优先考虑手机体验，但保留桌面可用性
5. **UI 极简**：白底、浅灰背景、基础卡片、普通导航，已有基础主题系统
6. **先跑通再打磨**：每次只做一小块，保持代码简单清晰
7. **联动检查**：改一个业务能力时，主动搜索并同步所有相关页面和入口
8. **共享优先**：同一业务规则不重复编写，抽到 `features/` 或 `hooks/`

## 一、项目定位

**一路记 WayLog** 是一款个人旅行助手 App，定位为「旅行控制台 + 旅行手账」，而非旅游社区或营销平台。核心目标是解决旅行信息分散、行程难整理、旅途中难查看的痛点，为个人用户提供一站式的旅行规划与管理体验。

- **Bundle ID**: `com.travelapp.mobile`
- **App 名称**: 一路记
- **Slug**: travel-app
- **版本**: 2026.06.10-1813
- **阶段**: MVP 迭代阶段

## 三、核心功能（当前已实现）

### 3.1 旅行管理
- ✅ 旅行 CRUD：创建、查看列表、编辑、删除行程
- ✅ 行程详情页：完整的行程信息展示
- ✅ 三种行程状态：计划中 / 旅途中 / 已完成
- ✅ 行程置顶功能

### 3.2 按天行程
- ✅ 按日期组织每日行程（TripDay）
- ✅ 每日包含多个日程项（TripDayItem）：时间、地点、备注、花费
- ✅ 时间排序与时间选择器
- ✅ 今日路线 Pills：横向标签链展示今日行程概览（地点数量、今日花费、预计结束时间）
- ✅ 当前地点高亮：基于时间判断当前应到达的地点

### 3.3 地点收藏与搜索
- ✅ 地点收藏（TripPlace）：26 种图标类型（景点/餐厅/酒店/交通/购物/教育/医疗等）
- ✅ 地点搜索：集成高德 POI 搜索
- ✅ 地点详情：名称、分类、坐标、地址、评分、照片、营业时间等
- ✅ 高德 v5 POI 富数据：评分、人均消费、图片等丰富信息
- ✅ 地点照片支持（来源：高德、Unsplash、Pixabay 等）
- ✅ 外部图片 API：集成 Unsplash + Pixabay API，为 POI 详情页提供额外图片资源
- ✅ 地图边界（AOI 多边形）：支持兴趣面绘制

### 3.4 AI 智能简介
- ✅ LLM 深度简介：通过 DeepSeek 大模型为景点生成旅行风格介绍
- ✅ 四层缓存策略：内存缓存 → AsyncStorage → 云端 POI 共享缓存 → Edge Function 生成
- ✅ 云端共享：生成的简介写入 poi_cache，所有用户共享，减少重复调用
- ✅ 智能过滤：自动跳过低价值地点（交通站点、卫生间等）

### 3.5 预算与记账

- ✅ 费用记录（TripExpense）：标题、金额、分类、日期、关联地点

## 四、数据模型

### 核心聚合根：Trip（行程）

```
Trip
├── TripDay[]              → 按天组织的日程
│   └── TripDayItem[]      → 单个行程点（时间/地点/备注/花费/推荐理由）
├── TripPlace[]            → 地点库（名称/分类/坐标/图标/照片/评分/详情/LLM简介/外部图片）
├── TripTransport[]        → 交通安排（航班/火车/自驾/巴士）
├── TripLodging[]          → 住宿安排（入住/退房/地址）
├── TripMemo[]             → 备忘（标题/详情/置顶）
├── TripChecklistItem[]    → 打包清单（标题/完成状态）
├── TripExpense[]          → 费用记录（分类/金额/日期/关联地点/支付人）
├── TripBudget             → 预算（金额/货币）
├── TripImportSource[]     → 导入来源（链接/文字/图片，解析待实现）
└── routeModeOverrides     → 路线交通方式手动覆盖
```

### 地点详情扩展（TripPlaceDetails）

```
TripPlaceDetails
├── summary?           → 地点简介（可由 LLM 生成）
├── highlights?        → 亮点/特色
├── cautions?          → 注意事项
├── openingHours?      → 开放时间
├── phone?             → 联系电话
├── website?           → 官网
├── ticketInfo?        → 门票信息
├── visitDuration?     → 建议游览时长
├── rating?            → 评分 (1-5)
├── ratingSource?      → 评分来源
├── reviewCount?       → 评论数
├── priceLevel?        → 消费水平（¥/¥¥/¥¥¥）
├── plannedCount?      → 计划去的人数
└── visitedCount?      → 已去过的人数
```

### 地点扩展类型

```
TripPlaceLLM           → LLM 生成内容（text）
TripPlaceExternalImage → 外部图片（Unsplash/Pixabay：id/url/source/author）
TripPlaceMapBoundary   → 地图边界（AOI 多边形坐标数组）
TripPlaceRef           → POI 引用（云端轻量存储，关联 poi_cache）
```

### 云端表结构（Supabase）

| 表名 | 用途 | 关键字段 |
| :-- | :-- | :-- |
| `public.profiles` | 用户资料 | display_name, avatar_url, bio, home_city |
| `public.user_trips` | 旅行数据 | user_id, payload(JSONB), version, deleted_at |
| `public.user_favorite_places` | 收藏地点 | user_id, payload(JSONB), provider_place_id, version, deleted_at |
| `public.user_route_preferences` | 路线偏好 | user_id, payload(JSONB) |
| `public.user_expense_preferences` | 费用偏好 | user_id, payload(JSONB) |
| `public.poi_cache` | POI 公共缓存 | amap_poi_id(主键), name, address, latitude, longitude, category, details(JSONB), photos(JSONB), externalRefs(JSONB) |
| `public.user_preferences` | 用户偏好 | user_id, payload(JSONB) |

> 所有表均启用 RLS（Row Level Security），通过 `auth.uid() = user_id` 进行数据隔离。poi_cache 表所有人可读、登录用户可写。

## 五、架构设计

### 5.1 分层架构

```
app/          → 页面编排、路由、屏幕级交互（Expo Router 文件路由）
components/   → UI 呈现、可复用视图（不写业务逻辑，纯展示组件）
features/     → 业务规则、数据存储、API 调用、纯逻辑（不依赖 React，可独立测试）
hooks/        → 应用级 React hooks
constants/    → 主题常量、设计令牌
```

### 5.2 数据流

```
UI (app/) → storage.ts (CRUD) → AsyncStorage (本地优先写入)
                               → cloud-sync.ts (后台异步同步) → Supabase
                               → poi-cache.ts (POI 公共缓存同步)
                               → llm.ts (AI 简介生成)
```

- **本地优先**：所有写操作先写入 AsyncStorage
- **脏标记**：写入后标记 dirty，触发后台同步
- **增量同步**：使用 upsert + 软删除，失败保留 dirty 标记下次重试
- **实时同步**：Supabase Realtime WebSocket 推送 + 页面聚焦刷新 + 定时轮询

### 5.3 POI 数据架构

```
高德 API → place-search.ts → TripPlace (本地)
                           → poi-cache.ts → poi_cache (云端共享)
                           → llm.ts → generate-place-info (Edge Function) → DeepSeek API
                                     → poi_cache (LLM 简介写回共享缓存)
                           → image-api.ts → Unsplash / Pixabay (外部图片)
```

### 5.4 路由结构

```
RootLayout
└── RootNavigation (Stack)
    ├── (tabs)                → 底部 Tab 导航
    │   ├── index             → 行程列表（首页）
    │   ├── create            → 新建行程（+号）
    │   └── profile           → 我的
    ├── account               → 账号与安全
    ├── appearance            → 外观设置（主题色 + 深色/浅色模式）
    ├── feedback              → 用户反馈
    ├── preferences           → 偏好设置
    ├── privacy               → 隐私政策
    ├── about                 → 关于页面
    ├── search                → 全局搜索
    ├── trips/new             → 新建行程
    ├── trips/[id]            → 行程详情（动态路由）
    ├── trips/edit            → 编辑行程
    ├── trips/place           → 地点详情
    ├── trips/favorite-place  → 收藏地点搜索

```

## 七、开发约定

1. **分层严格**：`app/` 只管页面编排，`components/` 只管 UI 呈现，`features/` 管业务逻辑和数据
2. **features 不依赖 React**：业务逻辑写成纯函数，可独立测试
3. **新增功能流程**：先在 `features/` 写逻辑和测试 → 再在 `app/` 和 `components/` 接入
4. **移动端优先**：优先考虑手机体验，但保留桌面可用性
5. **UI 极简**：白底、浅灰背景、基础卡片、普通导航，已有基础主题系统
6. **先跑通再打磨**：每次只做一小块，保持代码简单清晰
7. **联动检查**：改一个业务能力时，主动搜索并同步所有相关页面和入口
8. **共享优先**：同一业务规则不重复编写，抽到 `features/` 或 `hooks/`
9. **诊断日志**：使用 `createDiagnosticLogger` 输出结构化日志，支持重复折叠和敏感信息脱敏
